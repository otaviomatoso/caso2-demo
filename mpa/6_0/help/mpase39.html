C.2 Definição de Classes de Equipamentos

Classes de equipamentos são definidas através do construtor class. Cada definição indica o nome da classe de equipamentos e um identificador. Sugere-se que esse identificador siga as mesmas regras de identificadores de linguagens como Lua, C/C++ ou Java para permitir chamá-los de outras funções de forma mais usual. Além disso, os identificadores em letras maiúsculas são reservados par uso do MPA (e.g. identificadores de tipos básicos e tipos de pontos de controle).

A definição de uma classe de equipamentos também inclui uma lista de atributos, contendo o nome, o identificador e o tipo de valor de cada atributo. Os tipos de valores de atributos podem ser um tipo básico (BOOLEAN, INTEGER, REAL ou STRING), assim como um tipo de ponto de controle (BOOLEAN_POINT, INTEGER_POINT, REAL_POINT ou STRING_POINT) ou inclusive um outro equipamento, que é especificado através do identificador da classe do equipamento, e.g. ValvulaOnOff. Opcionalmente, os atributos também podem definir o campo access cujo valor é uma string contendo caracteres que definem as permissões de acesso dos diagramas ao atributo. Por exemplo, se um atributo define o campo access como ”gs” então os diagramas serão capazes de obter (”g” de get) e alterar (”s” de set) o valor do atributo. Se o campo access não é definido então o atributo não é acessível diretamente no diagrama.

Vale ressaltar que quando o tipo de um atributo é uma classe de ponto como REAL_POINT então a operação de obter (”g”) resulta no objeto que representa o ponto e não no valor do ponto no sistema supervisório. De forma similar, a operação de alteração (”s”) pode ser usada para mudar o ponto referenciado (i.e. o tag a ser acessado) e não alterar valor do ponto mantido pelo sistema supervisório. Por essa razão, o MPA define que atributos cujo tipo sejam classes de ponto podem definir modos de acesso adicionais. Por exemplo, um atributo com o campo access definido como ”rw” permitem que os diagramas possam ler (”r” de read) e escrever (”w” de write) o valor do ponto no supervisório.

Uma classe de equipamento também pode definir um conjunto de funções destinadas a permitir a manipulação dos equipamentos, como por exemplo realizar operações complexas que exijam manipular de forma sistemática vários pontos de controle relacionados ao equipamento (e.g. pulsar uma válvula choke). A definição de uma função de equipamento é composta por um nome, um identificador, uma descrição, uma lista de parâmetros e uma lista de resultados. Tanto a lista de parâmetros quanto a lista de resultados são arrays definindo o nome e o tipo de cada um dos parâmetros ou resultados. Os tipos dos parâmetros e valor de retorno das funções só podem ser tipos básicos ou equipamentos (definido através do identificador da classe de equipamento). Para parâmetros, o desenvolvedor por definir um valor padrão para um parâmetro atribuindo um valor ao campo default.

Opcionalmente, uma classe de equipamento pode definir uma lista de classes base das quais herdará as definições de atributos e métodos. Isso é feito através do campo bases que deve ser uma lista dos identificadores das classes bases. Alternativamente, uma classe pode ser definida como abstrata. Classes abstratas são usadas para definir comportamentos comuns para suas subclasses. Entretanto, para que esses comportamentos sejam utilizados, essas subclasses que devem extender a classe abstrata e ter instâncias definidas na configuração de planta. Para que uma classe seja considerada abstrata, o desenvolvedor do modelo desta classe deve definir o campo abstract com o valor true. Dessa forma, outras classes poderão usá-la como base e programar apenas o comportamento específico que corresponde a estas, cabendo à IDE impedir o desenvolvedor MPA de instânciá-la.

A figura C.1 ilustra a definição da classe Válvula Choke.


Figura C.1: Exemplo de definição de classe de equipamento.
 
class{ 
  id = ”ValvulaChoke”, 
  name = ”Valvula Choke”, 
  description = ”Válvula que permite pulsar sua abertura e seu fechamento”, 
 
  bases = { ”Valvula” }, -- a classe ’Valvula’ tem que ser definida também 
 
  attributes = { 
    { id = ”indicacaoPosicao”, 
      name = ”Indicacao posicao”, 
      type = ”REAL_POINT”, 
      access = ”r” }, 
    { id = ”comandoPulsoAbertura”, 
      name = ”Comando pulso abertura”, 
      type = ”BOOLEAN_POINT” }, 
 
    ... 
  }, 
 
  methods = { 
    { id = ”pulsarAbertura”, 
      name = ”Pulsar abertura”, 
      parameters = { 
        { name = ”Quantidade de pulsos”, type = ”INTEGER” }, 
        { name = ”Intervalo entre os pulsos”, type = ”REAL”, default = 10 }, }, 
      description = ”Executa seqüência de pulsos de abertura.”, 
      code = [[function(self, n, t) 
        while (n > 0) do 
          self.comandoPulsoAbertura = true 
          sleep(t) 
          n = n - 1 
        end 
      end]] }, 
 
    ... 
  }, 
}

A implementação de classes também pode definir operações internas que só são utilizadas por funções ou outras operações definidas no arquivo de módulo. Neste caso, a implementação dessas operações é feita através do campo code da classe, que pode adicionar ao campo _CLASS novos valores que serão automaticamente disponibilizados para todas as instâncias daquela classe. Por exemplo, para definir uma operação interna que calcula um número randômico com distribuição normal pode-se usar o trecho ilustrado na figura C.2.


Figura C.2: Exemplo de definição de classe de equipamento com campo code.
 
class{ 
  id = ”MinhaClasse”, 
  code = [[ 
    function _CLASS:normal() 
      local s = 2 
      while s >= 1 do 
        local u1 = random() 
        local u2 = random() 
        v1 = 2.0 * u1 - 1.0 
        v2 = 2.0 * u2 - 1.0 
        s = v1 * v1 + v2 * v2 
      end 
      return v1 * sqrt((-2.0 * log(s)) / s) 
    end 
  ]], 
  attributes = { ... }, 
  methods = { ... }, 
}

Alternativamente, também é possível usar o identificador da classe ao invés do nome _CLASS. O campo code também deve ser usado para implementar as operações declaradas através do campo methods.
